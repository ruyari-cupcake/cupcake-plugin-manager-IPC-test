🏗️ 4. 아키텍처 & 설계 패턴 6.5 / 10
강점
  • IPC 멀티플러그인 아키텍처 — Manager + 7 Sub-plugins, 각각 독립 iframe에서 실행. HELLO→ACK→FETCH→RESPONSE→ABORT 5단계 프로토콜
  • 공유 모듈 설계 — 8개 공통 모듈이 provider 간 코드 중복 제거. Rollup으로 각 번들에 인라인
  • 멀티 전략 Fetch — Strategy 1(bridge) → 2(nativeFetch/proxy2) → 3(risuFetch) 순차 폴백
  • 스트림 호환성 감지 — checkStreamCapability()로 structured-clone 가능 여부 런타임 판별
  • 키 회전 (KeyPool) — 라운드로빈 + Fisher-Yates 셔플 + drain 제거. 최대 30회 재시도
  • 지연 등록 지원 — 프로바이더가 늦게 등록해도 즉시 수용. ACK 미수신 시 지수 백오프 재시도 (12회)
  • 슬롯 추론 — 번역/감정/메모리/기타 슬롯을 요청 컨텍스트에서 자동 판별
  • Vertex AI OAuth 구현 — JWT + RSA-PKCS1 서명 via Web Crypto, 토큰 캐싱
약점
  • God Object: manager/index.js에 6개 이상의 책임이 혼재 — IPC 제어, 모델 등록, 요청 라우팅, 커스텀 모델 관리, 설정 UI, API 로깅
  • DI(의존성 주입) 없음 — 모듈 레벨 변수(registeredProviders, ALL_DEFINED_MODELS)에 의존하는 전역 상태
  • Feature 파일 모놀리식 — HTML·CSS·JS 혼합 (copilot.js, transcache.js, navigation.js)
  • 인터페이스 계약 미정의 — Provider 기능·메시지 타입 형식적 미정의
  • 에러 처리 비일관 — 일부는 에러 객체 반환, 일부는 throw, 일부는 console.error만
  • RisuAI V3 API 긴밀 결합 — 추상화 레이어 없이 직접 API 호출

AP-03. 하드코딩된 모델 목록 — 동기화 부재
각 프로바이더(anthropic.js, openai.js 등)가 모델 ID와 이름을 하드코딩합니다. AI 서비스 제공자가 모델을 추가/폐지할 때 수동으로 각 파일을 업데이트해야 하며, 동적 모델 조회 메커니즘이 없습니다.
📍 providers/*.js · 7개 파일 · 모델 정의 ~50개
AP-04. 멀티 전략 Fetch에서 항상 실패하는 Strategy 1
streamingFetch의 Strategy 1(직접 fetch)은 V3 iframe의 connect-src 'none' CSP로 인해 모든 플랫폼에서 100% 실패합니다. Session 6 감사(BUG-S6-2)에서 이미 보고된 문제이며 수정 완료되었으나, smartFetch에는 유사 패턴이 잔존할 수 있습니다.
📍 shared/helpers.js · streamingFetch / smartFetch
AP-05. 이중 Sanitize 호출 (성능 안티패턴)
7개 모든 프로바이더에서 sanitizeMessages()를 호출한 후 formatTo*()에 전달하는데, formatTo*() 내부에서도 sanitize를 재호출합니다. 대화 이력이 길 경우 O(2n) 불필요 처리가 발생합니다.
📍 providers/*.js 전체 · Session 6(BUG-S6-4)에서 수정 완료 확인 필요

📝 종합 평가 — 코드베이스 성격과 최종 판정

코드베이스 성격: CPM v4는 RisuAI V3 플러그인 시스템을 위한 멀티 프로바이더 AI API 게이트웨이입니다. 7개 AI 서비스(Anthropic, OpenAI, Gemini, Vertex AI, AWS Bedrock, DeepSeek, OpenRouter)를 IPC 기반 아키텍처로 통합하며, 커스텀 모델 정의, GitHub Copilot 토큰 관리, 번역 캐시, 채팅 UI 확장 등 부가 기능을 포함합니다. 한 명의 개발자가 구축한 것으로 보이며, 총 6,643줄의 JavaScript로 구성됩니다.

아키텍처적 강점: IPC 멀티플러그인 설계는 이 코드베이스의 가장 큰 자산입니다. Manager와 7개 Sub-plugin이 각각 독립 iframe에서 실행되며, HELLO/ACK/FETCH/RESPONSE/ABORT 5단계 프로토콜로 통신합니다. 공유 모듈 8개의 추출은 코드 중복을 효과적으로 방지하며, 멀티 전략 Fetch (bridge → proxy → risuFetch)는 V3 샌드박스의 CSP 제약을 우아하게 우회합니다. KeyPool의 키 회전·드레인 메커니즘, checkStreamCapability의 런타임 호환성 감지, Vertex AI의 Web Crypto 기반 JWT/OAuth 구현 등은 상당한 기술적 심도를 보여줍니다.

치명적 약점: 이 프로젝트의 근본적 문제는 엔지니어링 인프라의 완전한 부재입니다. 6,643줄의 프로덕션 코드에 테스트가 단 1개도 없으며, CI/CD, 린터, 타입 체커 등 어떤 품질 게이트도 구성되어 있지 않습니다. 이전 6차례의 자체 감사에서 발견된 21개 버그(CRITICAL 2건 포함)가 전부 수동 코드 리뷰로만 발견되었다는 것은, 자동화된 퇴행 방지 장치 없이 복잡한 메시지 변환·SSE 파싱·IPC 프로토콜이 운영되고 있음을 의미합니다. manager/index.js의 God Object 문제(2,182줄, 6개 이상 책임)는 유지보수를 더욱 어렵게 합니다.

보안 고려: RisuAI V3 샌드박스(iframe, connect-src 'none' CSP)는 많은 전통적 웹 취약점의 실질적 영향을 크게 완화합니다. 그러나 Gemini API 키의 URL 노출(V-01)은 호스트 앱 네트워크 계층에서의 키 유출 가능성을 야기하며, innerHTML 기반 UI 구성(V-02)은 입력값 검증이 불완전할 경우 이론적 XSS 통로입니다. AWS V4 서명의 자체 구현(V-03)은 Web Crypto API 기반으로 암호학적 기반은 견고하나, 커스텀 코드의 정확성 검증이 필요합니다.

개발 문화: 6건의 자체 감사 보고서와 21건의 체계적 버그 추적(BUG-Q*, BUG-S6-*)은 개발자의 품질 의식이 높음을 보여줍니다. 그러나 이 디버깅 노력이 수동 프로세스에 의존하고 있어, 자동화된 테스트로 전환하면 효율이 크게 향상될 것입니다.
